| **3** | **SELECT‑ы констант** (`_Const…`) | 264 вызова, ≈ 0,6 ms каждый; микросекундный overhead, не влияет на общую задержку. |
| **4** | **Отсутствие блокировок/таймаутов** (Linux/PG) | `linux.has_cpu_pressure = false`, `postgres_runtime.has_pg_waits = true` (но только клиентские/вакуум‑ожидания), `deadlocks.events = []`. |
| **5** | **Недостаток памяти** | `linux.mem_available_pct = 34 %` – достаточный запас, нет swap‑давления. |
| **6** | **SQL‑plan проблемы** (по ресурс‑корреляции) | Плановые проблемы (Seq Scan, temp‑spill) подтверждены в 06_postgresql, но они относятся к **query‑design**, а не к аппаратному ограничению. |

## 7. Ограничения анализа  

| Ограничение | Описание |
|-------------|----------|
| **Объём данных** | Анализ выполнен на основе предоставленных partial reports; не включены полные журналы запросов, трассировки кода и полные дампы планов. |
| **Отсутствие полной корреляции CALL ↔ SQL** | В payload нет прямой привязки `sql_query_id` к каждому CALL, поэтому часть гипотез (например, точный вклад отдельных SCALL) остаётся непроверенной. |
| **Отсутствие метрик блокировок** | `lock_findings` пуст, но это может быть следствием отключённого сбора; реальное наличие блокировок не исключено. |
| **Не учитываются внешние сервисы** | Сетевые запросы и COM‑компоненты не измеряются в текущем наборе метрик, поэтому их влияние оценено только гипотетически. |
| **Только один временной интервал** | Метрики собраны за ограниченный период; динамика нагрузки (пиковые часы) может отличаться. |
| **Отсутствие информации о конфигурации PostgreSQL** | Параметры `work_mem`, `shared_buffers`, `effective_io_concurrency` не известны, поэтому рекомендации по их изменению носят предварительный характер. |
| **Не проверена полнота логирования** | Отсутствие записей в `exceptions`, `deadlocks`, `timeouts` может быть следствием низкого уровня логирования, а не отсутствия проблем. |

---  

**Итог:**  
Наиболее критичными узкими местами являются **тяжёлые прикладные CALL без SQL** и **неэффективные запросы к таблице `Config`**, подкреплённые **wait‑событиями PostgreSQL**. Предлагаемый план дальнейшего анализа и набор рекомендаций направлен на изоляцию проблемных участков кода, улучшение планов запросов и устранение потенциальных I/O‑проблем, что должно привести к заметному снижению времени отклика пользовательских форм и стабилизации фоновых процессов.

## 24. Прочие ограничения анализа

- Cross-source корреляция зависит от степени унификации normalized SQL между 1С ТЖ и PostgreSQL.
- Для кейса `Config WHERE FileName LIKE '%'` индекс по `FileName` помогает только при селективном префиксном шаблоне; при `LIKE '%'` ключевой путь оптимизации — устранение полного чтения, кэширование или изменение сценария обращения.
